# SIOT Widget Dashboard Design System Governance (AI Context)

> 이 문서는 AI(Claude/Gemini 등)가 프로젝트 작업 시 반드시 따라야 하는 절대 규칙과 컨텍스트입니다.
> 시스템 프롬프트(User Rules)에 주입되어 매 작업 시 최우선으로 검토됩니다.

---

## 🎨 1. 디자인 시스템 규칙 (최우선!)

> ⚠️ **모든 UI 작업은 아래 디자인 시스템을 기반으로 합니다. 예외 없음.**

### 토큰 파일 구조 (단일 소스 동기화 필수)

| 파일 | 역할 | 수정 기준 |
|------|------|-----------|
| `design-tokens.json` | 토큰 원본 정의 (Hex, Px, rem 값은 **여기에만** 존재) | 신규 토큰 필요 시 |
| `index.css` | JS/React에서 참조할 CSS 변수(`--var`) 등록 | json 업데이트 시 동기화 |
| `themeFromTokens.ts` | CSS 변수를 React 테마 객체로 변환하여 주입 | 구조 변경 시 |

### ⛔ 디자인 시스템 금지사항 (코드 작성 시 체크리스트)

| 금지 | 올바른 방법 | 예시 |
|------|-------------|------|
| **헥스 코드/Px 직접 사용** | CSS 변수(`var(--)`) 사용 | `#3b82f6` ❌ → `var(--primary-color)` ✅ |
| **Tailwind 임의값 사용** | 커스텀 토큰 혹은 기본 유틸 | `bg-black/60` ❌ → `bg-[var(--export-overlay-backdrop-bg)]` ✅ |
| **토큰에 없는 spacing** | 토큰 값만 사용 | `p-[15px]` ❌ → `p-4` 혹은 토큰 추가 제안 ✅ |
| **기존 컴포넌트 중복 생성** | 기존 디자인 컴포넌트 재사용 | 직접 `<div>`로 버튼 만들기 ❌ → `<Button>` 재사용 ✅ |
| **사전 허락 없는 토큰 추가** | 승인 후 3파일 동시 업데이트 | `index.css`만 임의 수정 ❌ → 제안 후 json/css 동시 수정 ✅ |

## 📝 2. 새 토큰 추가 프로세스

> 기존 토큰이 요구사항에 맞지 않거나, 새로운 UI 컴포넌트(Checkboxes, Radio 등)를 추가할 때 적용됩니다.

**1단계: Propose (제안)**
*   어떤 토큰이 왜 필요한지 사용자에게 먼저 제안하고 승인(`Approve`)을 받습니다.

**2단계: Update JSON & CSS (적용)**
*   `design-tokens.json`에 토큰을 추가합니다.
*   `index.css`에 CSS 변수를 매핑합니다.

**3단계: Document (문서화)**
*   필요 시 `DesignDocs.tsx` 등에 토큰 내역을 업데이트합니다.

**4단계: Implement (구현)**
*   실제 컴포넌트 코드(`TSX`)에 토큰을 적용하여 개발합니다.

## 🤖 3. AI 작업 프로세스 및 원칙 (코드 작성 전 필수 보고)

### 기본 작업 원칙
- **확장성/유연성 검토**: 현재 요구사항을 해결하되, 향후 확장이 막히지 않는 구조를 확인합니다.
- **기존 코드 재사용**: 새로 만들기 전에 `components/` 등 기존 리소스를 먼저 탐색합니다.
- **커뮤니케이션**: 항상 **개요(왜, 무엇을)** → **상세 구현 계획** 순서로 설명합니다.

### 작업 프로세스 4단계 (필수!)

> ⚠️ **추측을 사실처럼 말하지 말 것!** 모든 가설은 반드시 검증 후 결론을 도출합니다.
> ⛔ **코드 작성 전 반드시 아래 4단계까지 완료하고 사용자 승인을 받을 것.**

#### 1단계: 문제/요청 이해
- 문제 현상을 명확히 기술합니다.
- 불분명한 부분이 있으면 사용자에게 질문합니다.
- "~일 것 같습니다"가 아니라 실제 코드를 확인하여 팩트 기반으로 접근합니다.

#### 2단계: 원인 분석 (문제 해결의 경우)
- 가설 수립 → 가설 검증 → 원인 확정
- ❌ "이게 원인입니다" (검증 없이 단정 짓는 것 금지)
- ✅ "가설: ~일 수 있습니다. 코드를 확인해 보겠습니다." → "확인 결과, ~가 원인입니다."

#### 3단계: 해결책 탐색
- 해결 방안 2~3개 제시, 각 방안의 영향 범위 분석
- 사전 검증 가능하면 검증

#### 4단계: 작업 계획 보고 (코드 작성 전 필수!)

> ⛔ 사용자가 "그냥 빨리 해줘"라고 해도, **반드시 이 보고 양식을 먼저 제출**하세요.

```text
📋 작업 계획 보고

🔍 문제 상황 (What's wrong?)
어떤 상황에서 어떤 증상이 발생하는지, 왜 이 작업이 필요한지.

🎯 목표 (What we want to achieve)
이 작업이 완료되면 어떤 상태가 되어야 하는지.

🔬 원인 분석 (Why it happens) - 문제 해결의 경우
검증된 원인만 기술. 추측은 "가설"이라고 명시.

📁 변경 예정 파일
| 파일 경로 | 변경 내용 | 비고 |
|-----------|----------|------|

⚡ Before → After
[Before] 현재 상태
[After] 작업 후 기대 상태

🎨 디자인 토큰 사용 계획
- 사용할 CSS 변수: --primary-color, --spacing-md 등
- 재사용할 컴포넌트: WidgetCard 등
- 새로 필요한 토큰: 있으면 명시 (없으면 "없음")

이대로 진행해도 될까요?
```

## 🔬 4. 결과 검증 체크리스트 (작업 완료 선언 전)

AI는 "작업을 완료했습니다"라고 말하기 전에 스스로 아래 항목을 체크해야 합니다.

- [ ] 신규 작성 코드에 하드코딩된 값(Hex, Px 등)이 없는가?
- [ ] Tailwind 임의값(`w-[...]`) 대신 토큰 변수를 사용했는가?
- [ ] 토큰 추가 시 SSoT 구조(`json` -> `css`)를 모두 동기화했는가?
- [ ] 컴포넌트 내부 스타일을 직접 수정하지 않고 props로 제어했는가?
- [ ] 빌드 에러 및 기존 레이아웃 깨짐 현상이 없는가?

## 🐙 5. Git 협업 규칙 (Git Governance)

- **무단 Push 금지**: AI 어시스턴트는 로컬 커밋(`git commit`)까지는 자유롭게 진행할 수 있으나, 원격 저장소(`origin main`)로의 **Push는 반드시 사용자의 명시적인 허락**을 받은 후 실행합니다.
- 작업 완료 후 항상 "수정 사항을 Git에 반영할까요?"라고 승인을 요청합니다.
- 커밋 메시지는 반드시 `GIT_CONVENTION.md`에 따라 **한국어**로 작성합니다.

## 🛠️ 7. 기술 스택 (Tech Stack)

- **언어**: TypeScript (`.ts`, `.tsx`) - **모든 새 코드는 반드시 TS 기반으로 작성**
- **스타일링**: CSS Variables (`index.css`) + Tailwind CSS
- **토큰 관리**: `design-tokens.json` 기반의 커스텀 테마 엔진 (`themeFromTokens.ts`)
- **컴포넌트**: React (함수형 컴포넌트 + Hooks)
- **레이아웃**: `react-grid-layout` 기반의 대시보드 그리드 시스템
- **데이터베이스/인증**: Supabase

## ⛔ 8. 공통 금지 사항 (Strict Prohibitions)

| 금지 | 이유 | 올바른 대안 |
|------|------|-------------|
| **허락 없이 새 파일/컴포넌트 생성** | 프로젝트 구조 임의 변경 방지 | 사용자에게 먼저 제안 후 승인 |
| **기존 아키텍처 임의 변경** | 설계 의도 훼손 방지 | 변경 필요 시 이유와 함께 제안 |
| **요청 범위 밖 리팩토링** | 스코프 크립(Scope Creep) 방지 | "이 부분도 개선하면 좋겠는데, 할까요?" 사전 확인 |
| **문제 발견 시 바로 수정** | 사용자가 다른 해결책을 원할 수 있음 | 문제 보고 → 해결책 2~3개 제시 → 승인 후 수정 |
| **디자인 토큰 없이 스타일링** | 디자인 시스템 일관성 파괴 | 항상 `var(--*)` 또는 Tailwind 커스텀 토큰 사용 |

## 🔄 9. 이전 세션 작업 이어받을 때 (Session Resumption)

이전 대화를 바탕으로 프로젝트를 이어갈 때, AI는 다음 원칙을 반드시 지켜야 합니다.

1. **"완료됐다"는 요약을 그대로 믿지 말 것** (이전 세션의 환각일 수 있음)
2. **실제 코드 상태를 직접 확인** (파일 읽기, 검색 도구 등을 활용하여 팩트 검증)
3. **무작정 코드 작성 금지** (현재 구조를 정확히 파악하고 검증한 뒤에 작업을 시작할 것)
